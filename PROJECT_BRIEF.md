# PROJECT_BRIEF.md

Суммаризация требований к проекту «Запись на звонок» для будущих сессий.
Дополняется по мере развития проекта. Технические детали репозитория — в `AGENTS.md`.

## 1. Суть проекта
Упрощённый аналог Cal.com: сервис бронирования времени для звонка.
- Владелец календаря публикует типы событий и доступные интервалы.
- Гость выбирает свободный слот и бронирует встречу.
- Нет регистрации, авторизации, личных кабинетов, внешних интеграций (календари, email и т.д.).

## 2. Роли
- **Владелец календаря** — один заранее заданный профиль (hardcoded). Используется по умолчанию в админской части.
  - Создаёт типы событий: `id`, `name`, `description`, `durationMinutes`.
  - Смотрит страницу предстоящих встреч: единый список бронирований всех типов событий.
  - Идентификация: `VITE_ADMIN_SECRET` в `.env`. Админка доступна по `/admin`.
- **Гость** — без аккаунта и входа.
  - Видит список типов событий (название, описание, длительность).
  - Выбирает тип события → открывает календарь → выбирает свободный слот.
  - Создаёт бронирование на выбранный слот.

## 3. Бизнес-правила
- **Слоты**: шаг = `durationMinutes` из EventType (15 или 30 минут).
- **Окно записи**: 14 дней, начиная с текущей даты.
- **Занятость**: на одно и то же время не может быть двух бронирований, даже для разных типов событий (глобальная занятость по времени, не по типу).
- Гость может записаться только на свободный слот из окна 14 дней.
- **Рабочие часы**: 09:00–18:00 (хардкод в бэке, пока нет настроек).

## 4. Подход к разработке: Design First
1. Сначала фиксируется API-контракт в TypeSpec (`packages/typespec/main.tsp`).
2. Из TypeSpec генерируется OpenAPI (`apps/backend/openapi.yaml` — автогенерируемый, не править руками).
3. Из OpenAPI генерируются TS-типы для фронта (`apps/frontend/src/api/schema.ts`).
4. Контракт — **единый источник правды** для фронтенда и бэкенда.
5. Фронт и бэк реализуются **независимо** и взаимодействуют только через API по контракту.
6. При изменениях: сначала контракт → затем синхронные правки во фронт и бэк.

## 5. Инструменты разработки
- **TypeSpec** — описание API-контракта.
- **Prism** — мок-сервер по OpenAPI для разработки фронта без поднятого бэка.
- **Vite proxy** — перенаправляет `/api` на Prism (dev) или бэк (prod).
- **Фронтенд**: React + React Router + Mantine + Vite + TypeScript (`apps/frontend`).
- **Бэкенд**: .NET 8 + ASP.NET + EF Core + SQLite (`apps/backend`).
- Подробности команд и структуры репозитория — в `AGENTS.md`.

## 6. Роутинг фронтенда
- `/` — лендинг
- `/event-types` — гость: список типов событий
- `/event-types/:eventTypeId/slots` — гость: выбор даты и слота
- `/admin` — админка владельца (требует `VITE_ADMIN_SECRET`)

## 7. Конфигурация
- **API Base URL**: `VITE_API_TARGET` в `.env`
  - Dev: `http://localhost:3000` (Prism mock)
  - Prod: `http://localhost:5000` (реальный бэк)
- **Admin Secret**: `VITE_ADMIN_SECRET` в `.env`
- Фронт обращается к `/api/*`, Vite proxy решает, куда направить.

## 8. Генерация типов
```bash
npm run build:typespec  # генерирует openapi.yaml + schema.ts
```
- TypeSpec → OpenAPI (`apps/backend/openapi.yaml`)
- OpenAPI → TS-типы (`apps/frontend/src/api/schema.ts`)

## 9. Текущий контракт API (кратко)
Base: `/api` (проксируется на `VITE_API_TARGET`). Пути в kebab-case.
- `GET  /event-types` — список типов.
- `GET  /event-types/{eventTypeId}` — один тип.
- `POST /event-types` — создать тип (требует admin secret).
- `GET  /slots?eventTypeId&date` — доступные слоты на дату.
- `POST /bookings` — создать бронирование.
- `GET  /bookings` — список всех бронирований (требует admin secret).
- `DELETE /bookings/{bookingId}` — удалить бронирование.

Модели: `EventType`, `CreateEventType`, `Slot`, `Booking`, `CreateBooking`, `Error`.

> Известное расхождение (из `AGENTS.md`): TypeSpec эмитит kebab-case (`/api/event-types`), а .NET-контроллеры сейчас отвечают на `/api/EventTypes`, `/api/Bookings`, `/api/Slots`. При работе с API-роутами синхронизировать обе стороны.

## 10. Требования к реализации
- Фронт ходит в API **только** по контракту; никаких знаний о внутренностях бэка.
- Фронт должен корректно работать с отдельно запущенным бэкендом (или Prism-моком).
- Бэк реализует контракт полностью, не выходит за его пределы.
- Любое изменение поведения = изменение контракта в TypeSpec.

## 11. Валидация (фронт)
- Guest name: мин. 2 символа
- Guest email: стандартный email-паттерн
- Ошибки API: пока не реализованы (нотификации Mantine)
